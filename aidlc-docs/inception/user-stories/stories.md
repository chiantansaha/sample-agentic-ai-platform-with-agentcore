# 사용자 스토리

## 1. MCP 관리 스토리

### 1.1 MCP 조회 및 검색
**사용자 스토리**: MCP/KB 개발자로서, 등록된 MCP 목록을 조회하고 검색할 수 있어야 한다.

**승인 기준**:
- MCP 목록에서 MCP명, 엔드포인트, 설명, 버전, 툴 개수, 내부/외부 상태, 활성화/비활성화 상태를 확인할 수 있다
- MCP명으로 검색할 수 있다
- 내부/외부 상태로 필터링할 수 있다
- 활성화/비활성화 상태로 필터링할 수 있다
- 검색 결과가 없을 때 적절한 메시지가 표시된다

### 1.2 MCP 상세 조회
**사용자 스토리**: MCP/KB 개발자로서, 특정 MCP의 상세 정보를 조회할 수 있어야 한다.

**승인 기준**:
- MCP 개요 정보를 확인할 수 있다
- MCP에 포함된 툴 목록을 확인할 수 있다
- MCP 버전 정보를 확인할 수 있다
- MCP 연결 상태를 확인할 수 있다

### 1.3 외부 MCP 등록
**사용자 스토리**: MCP/KB 개발자로서, 외부 MCP를 플랫폼에 등록할 수 있어야 한다.

**승인 기준**:
- MCP명을 입력할 수 있다
- MCP 설명을 입력할 수 있다
- MCP 엔드포인트를 입력할 수 있다
- 인증 방식을 선택할 수 있다 (None, OAuth, API Key)
- 등록 완료 후 외부 게이트웨이에 Target으로 추가된다
- 등록 실패 시 오류 메시지가 표시된다

### 1.4 내부 MCP 등록
**사용자 스토리**: MCP/KB 개발자로서, 내부 MCP를 컨테이너 이미지로 등록할 수 있어야 한다.

**승인 기준**:
- MCP명을 입력할 수 있다
- MCP 설명을 입력할 수 있다
- 컨테이너 이미지를 업로드할 수 있다
- 등록 완료 후 런타임에 배포된다
- 업로드 진행 상황을 확인할 수 있다
- 업로드 실패 시 오류 메시지가 표시된다

### 1.5 내부 MCP 생성
**사용자 스토리**: MCP/KB 개발자로서, API 정보를 입력하여 내부 MCP를 생성할 수 있어야 한다.

**승인 기준**:
- MCP명을 입력할 수 있다
- MCP 설명을 입력할 수 있다
- API 엔드포인트를 입력할 수 있다
- API 스키마를 입력할 수 있다
- 인증 URL을 입력할 수 있다
- 생성 완료 후 게이트웨이가 생성된다

### 1.6 MCP 활성화/비활성화
**사용자 스토리**: MCP/KB 개발자로서, MCP를 활성화하거나 비활성화할 수 있어야 한다.

**승인 기준**:
- MCP 목록에서 활성화/비활성화 토글을 사용할 수 있다
- 상태 변경이 즉시 반영된다
- 비활성화된 MCP는 에이전트 생성 시 선택할 수 없다
- 상태 변경 실패 시 오류 메시지가 표시된다

### 1.7 MCP 버전 업데이트
**사용자 스토리**: MCP/KB 개발자로서, MCP를 새 버전으로 업데이트할 수 있어야 한다.

**승인 기준**:
- 기존 MCP의 새 버전을 업로드할 수 있다
- AWS AgentCore Gateway 버전 정책에 따라 버전이 관리된다
- 업데이트 진행 상황을 확인할 수 있다
- 업데이트 실패 시 이전 버전으로 롤백된다

## 2. 지식베이스 관리 스토리

### 2.1 지식베이스 조회 및 검색
**사용자 스토리**: MCP/KB 개발자로서, 등록된 지식베이스 목록을 조회하고 검색할 수 있어야 한다.

**승인 기준**:
- 지식베이스 목록에서 이름, 설명, 활성화/비활성화 상태를 확인할 수 있다
- 지식베이스명으로 검색할 수 있다
- 활성화/비활성화 상태로 필터링할 수 있다
- 검색 결과가 없을 때 적절한 메시지가 표시된다

### 2.2 지식베이스 생성
**사용자 스토리**: MCP/KB 개발자로서, 새로운 지식베이스를 생성할 수 있어야 한다.

**승인 기준**:
- 지식베이스명을 입력할 수 있다
- 지식베이스 설명을 입력할 수 있다
- 마크다운 파일(.md)을 업로드할 수 있다
- 업로드된 파일이 S3 버킷에 저장된다
- 벡터 임베딩이 생성되어 OpenSearch에 저장된다
- 처리 진행 상황을 확인할 수 있다
- 처리 실패 시 오류 메시지가 표시된다

### 2.3 지식베이스 데이터소스 수정
**사용자 스토리**: MCP/KB 개발자로서, 기존 지식베이스의 데이터소스를 수정할 수 있어야 한다.

**승인 기준**:
- 기존 파일을 새 파일로 교체할 수 있다
- 새 파일을 추가할 수 있다
- 기존 파일을 삭제할 수 있다
- 변경사항이 자동으로 동기화된다
- 동기화 진행 상황을 확인할 수 있다
- 동기화 실패 시 오류 메시지가 표시된다

### 2.4 지식베이스 비활성화
**사용자 스토리**: MCP/KB 개발자로서, 지식베이스를 비활성화할 수 있어야 한다.

**승인 기준**:
- 지식베이스 목록에서 비활성화 기능을 사용할 수 있다
- 비활성화된 지식베이스는 에이전트 생성 시 선택할 수 없다
- 비활성화 상태가 즉시 반영된다

## 3. 에이전트 관리 스토리

### 3.1 에이전트 조회 및 검색
**사용자 스토리**: 에이전트 개발자로서, 등록된 에이전트 목록을 조회하고 검색할 수 있어야 한다.

**승인 기준**:
- 에이전트 목록에서 이름, 설명, 활성화/비활성화 상태를 확인할 수 있다
- 에이전트명으로 검색할 수 있다
- 활성화/비활성화 상태로 필터링할 수 있다
- 검색 결과가 없을 때 적절한 메시지가 표시된다

### 3.2 에이전트 상세 조회
**사용자 스토리**: 에이전트 개발자로서, 특정 에이전트의 상세 정보를 조회할 수 있어야 한다.

**승인 기준**:
- 에이전트 개요 정보를 확인할 수 있다
- 연결된 툴 목록을 확인할 수 있다
- 연결된 지식베이스 목록을 확인할 수 있다
- 에이전트 버전 정보를 확인할 수 있다
- 배포 상태를 확인할 수 있다

### 3.3 에이전트 생성
**사용자 스토리**: 에이전트 개발자로서, 새로운 에이전트를 생성할 수 있어야 한다.

**승인 기준**:
- LLM 모델을 선택할 수 있다 (Bedrock 모델만)
- 에이전트 인스트럭션을 작성할 수 있다
- 지식베이스를 선택할 수 있다
- 외부 MCP를 선택할 수 있다
- 내부 MCP(게이트웨이)를 선택할 수 있다
- 에이전트 명세가 저장된다
- 저장 실패 시 오류 메시지가 표시된다

### 3.4 에이전트 배포 준비
**사용자 스토리**: 에이전트 개발자로서, 생성한 에이전트를 런타임에 배포할 수 있어야 한다.

**승인 기준**:
- 에이전트를 런타임에 배포할 수 있다
- 배포 진행 상황을 확인할 수 있다
- 배포 완료 후 채팅 기능이 활성화된다
- 배포 실패 시 오류 메시지가 표시된다

### 3.5 에이전트 수정
**사용자 스토리**: 에이전트 개발자로서, 기존 에이전트를 수정할 수 있어야 한다.

**승인 기준**:
- LLM 모델을 변경할 수 있다
- 인스트럭션을 수정할 수 있다
- 연결된 툴을 변경할 수 있다
- 수정사항이 저장되고 버전이 업데이트된다
- 수정 실패 시 오류 메시지가 표시된다

### 3.6 에이전트 재배포
**사용자 스토리**: 에이전트 개발자로서, 수정된 에이전트를 재배포할 수 있어야 한다.

**승인 기준**:
- 수정된 에이전트를 런타임에 재배포할 수 있다
- 재배포 진행 상황을 확인할 수 있다
- 재배포 완료 후 채팅에서 변경사항이 반영된다
- 재배포 실패 시 이전 버전으로 롤백된다

### 3.7 에이전트 비활성화
**사용자 스토리**: 에이전트 개발자로서, 에이전트를 비활성화할 수 있어야 한다.

**승인 기준**:
- 에이전트를 비활성화할 수 있다
- 비활성화된 에이전트는 플레이그라운드에서 선택할 수 없다
- 비활성화 상태가 즉시 반영된다

## 4. 플레이그라운드 스토리

### 4.1 에이전트 선택
**사용자 스토리**: 최종 사용자로서, 사용 가능한 에이전트 중에서 테스트할 에이전트를 선택할 수 있어야 한다.

**승인 기준**:
- 활성화된 에이전트 목록을 확인할 수 있다
- 각 에이전트의 기본 정보(이름, 설명)를 확인할 수 있다
- 에이전트를 선택할 수 있다
- 선택한 에이전트가 명확히 표시된다

### 4.2 에이전트 준비
**사용자 스토리**: 최종 사용자로서, 선택한 에이전트가 채팅 준비를 완료할 때까지 대기할 수 있어야 한다.

**승인 기준**:
- 에이전트 준비 상태를 확인할 수 있다
- 준비 진행 상황이 표시된다
- 준비 완료 시 채팅 인터페이스가 활성화된다
- 준비 실패 시 오류 메시지가 표시된다

### 4.3 채팅 인터페이스
**사용자 스토리**: 최종 사용자로서, 배포된 에이전트와 채팅할 수 있어야 한다.

**승인 기준**:
- 텍스트 메시지를 입력할 수 있다
- 메시지를 전송할 수 있다
- 에이전트의 응답을 확인할 수 있다
- 대화 기록이 유지된다
- 메시지 전송 실패 시 오류 메시지가 표시된다

### 4.4 에이전트 테스트
**사용자 스토리**: 최종 사용자로서, 다양한 질문으로 에이전트를 테스트할 수 있어야 한다.

**승인 기준**:
- 연속적인 대화를 진행할 수 있다
- 다양한 주제의 질문을 할 수 있다
- 에이전트가 일관된 응답을 제공한다
- 대화 컨텍스트가 유지된다
- 응답 시간이 합리적인 범위 내에 있다

### 4.5 새로운 채팅 세션 시작
**사용자 스토리**: 최종 사용자로서, 기존 대화를 종료하고 새로운 채팅 세션을 시작할 수 있어야 한다.

**승인 기준**:
- 새로운 채팅 세션을 시작할 수 있다
- 이전 대화 기록이 초기화된다
- 에이전트가 초기 상태로 리셋된다
- 새 세션에서 정상적으로 대화할 수 있다

### INVEST 기준 준수
- **Independent**: 각 스토리는 독립적으로 구현 가능
- **Negotiable**: 구현 방법에 대한 협상 가능
- **Valuable**: 각 스토리는 사용자에게 명확한 가치 제공
- **Estimable**: 구현 복잡도 추정 가능
- **Small**: 한 스프린트 내에서 완료 가능한 크기
- **Testable**: 명확한 승인 기준으로 테스트 가능
