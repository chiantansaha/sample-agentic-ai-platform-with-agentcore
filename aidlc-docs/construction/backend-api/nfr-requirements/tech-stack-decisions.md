# Backend API Service 기술 스택 결정사항

## 개요
NFR 요구사항을 기반으로 Backend API Service의 기술 스택을 선택하고 결정 근거를 문서화합니다.

## 웹 프레임워크

### 선택: FastAPI
**결정 근거**:
- **성능**: 비동기 처리로 높은 처리량 지원 (목표: 평균 1초 응답시간)
- **타입 안전성**: Pydantic을 통한 자동 검증 및 문서화
- **WebSocket 지원**: 실시간 채팅 기능 구현
- **자동 문서화**: OpenAPI/Swagger 자동 생성
- **Python 생태계**: AI/ML 라이브러리와의 호환성

**대안 고려사항**:
- Django: 너무 무겁고 성능 요구사항에 부적합
- Flask: 기능이 부족하고 확장성 제한
- Node.js: Python 생태계 이점 포기

## 데이터베이스

### 주 데이터베이스: Amazon DynamoDB
**결정 근거**:
- **확장성**: 온디맨드 모드로 자동 확장 (목표: 50 동시 사용자)
- **성능**: 50ms 이하 지연시간 목표 달성
- **가용성**: 99.99% SLA로 99.5% 목표 초과 달성
- **관리형**: 운영 부담 최소화
- **비용 효율성**: 사용량 기반 과금

**테이블 설계**:
- `workspaces`: 워크스페이스 메타데이터
- `mcp_servers`: MCP 서버 정보
- `knowledge_bases`: 지식베이스 메타데이터
- `agents`: 에이전트 정보
- `chat_sessions`: 채팅 세션 데이터
- `chat_messages`: 채팅 메시지 (TTL 적용 가능)

### 검색 엔진: Amazon OpenSearch Service
**결정 근거**:
- **벡터 검색**: 임베딩 기반 의미 검색 지원
- **성능**: 300ms 이하 검색 지연시간 목표
- **확장성**: 클러스터 자동 확장
- **관리형**: 인덱스 관리 자동화
- **통합성**: AWS 생태계 통합

**인덱스 설계**:
- `knowledge-chunks`: 지식베이스 청크 및 임베딩
- `chat-history`: 채팅 이력 검색용

## 파일 저장소

### 선택: Amazon S3
**결정 근거**:
- **확장성**: 무제한 저장 용량
- **내구성**: 99.999999999% (11 9's) 내구성
- **보안**: 암호화 및 접근 제어 지원
- **비용 효율성**: 스토리지 클래스별 최적화
- **통합성**: Lambda 트리거 지원

**버킷 구조**:
- `{workspace-id}/knowledge-bases/{kb-id}/files/`: 원본 파일
- `{workspace-id}/knowledge-bases/{kb-id}/chunks/`: 처리된 청크
- `{workspace-id}/agents/{agent-id}/artifacts/`: 에이전트 아티팩트

## 메시지 큐

### 선택: Amazon SQS + Amazon SNS
**결정 근거**:
- **비동기 처리**: 파일 처리 워크플로우 지원
- **확장성**: 자동 확장 및 부하 분산
- **신뢰성**: 메시지 전달 보장
- **통합성**: Lambda 트리거 지원
- **비용 효율성**: 사용량 기반 과금

**큐 설계**:
- `file-processing-queue`: 파일 처리 작업
- `embedding-queue`: 임베딩 생성 작업
- `indexing-queue`: 검색 인덱스 업데이트
- `notification-topic`: 상태 변경 알림

## 캐싱

### 선택: Redis (Amazon ElastiCache)
**결정 근거**:
- **성능**: 메모리 기반 초고속 액세스
- **세션 관리**: 분산 세션 저장소
- **캐싱**: API 응답 캐싱으로 성능 향상
- **실시간**: WebSocket 세션 관리
- **확장성**: 클러스터 모드 지원

**캐시 전략**:
- JWT 토큰 블랙리스트
- API 응답 캐싱 (TTL: 5분)
- 사용자 세션 정보
- 임시 파일 처리 상태

## 인증 및 권한

### 선택: JWT + AWS Cognito (선택적)
**결정 근거**:
- **표준성**: JWT 표준 준수
- **확장성**: 상태 비저장 토큰
- **보안**: 서명 검증 및 만료 시간
- **통합성**: 프론트엔드와 쉬운 통합
- **유연성**: 커스텀 클레임 지원

**토큰 설계**:
- **Access Token**: 1시간 만료
- **Refresh Token**: 24시간 만료
- **클레임**: user_id, workspace_id, roles

## 모니터링 및 로깅

### 선택: Amazon CloudWatch
**결정 근거**:
- **통합성**: AWS 서비스 네이티브 통합
- **확장성**: 자동 확장 및 보관
- **알림**: CloudWatch Alarms 지원
- **대시보드**: 실시간 모니터링
- **비용 효율성**: 기본 모니터링 무료

**메트릭 설계**:
- **애플리케이션**: 응답 시간, 처리량, 오류율
- **인프라**: CPU, 메모리, 네트워크
- **비즈니스**: 활성 사용자, 파일 처리 수

### 로깅: 구조화된 JSON 로깅
**로그 레벨**:
- ERROR: 시스템 오류 및 예외
- WARN: 경고 및 성능 이슈
- INFO: 주요 비즈니스 이벤트
- DEBUG: 상세 디버깅 정보

## 배포 및 인프라

### 컨테이너화: Docker
**결정 근거**:
- **일관성**: 환경 간 일관된 실행
- **확장성**: 컨테이너 오케스트레이션
- **격리**: 애플리케이션 격리
- **배포**: 빠른 배포 및 롤백

### 오케스트레이션: Amazon ECS
**결정 근거**:
- **관리형**: 서버리스 컨테이너 실행
- **확장성**: 자동 확장 및 부하 분산
- **통합성**: AWS 서비스 네이티브 통합
- **비용**: Fargate로 서버 관리 불필요

### CI/CD: GitHub Actions + AWS CodeDeploy
**파이프라인 단계**:
1. 코드 커밋 및 푸시
2. 자동 테스트 실행
3. Docker 이미지 빌드
4. ECR에 이미지 푸시
5. ECS 서비스 업데이트
6. 헬스 체크 및 롤백
